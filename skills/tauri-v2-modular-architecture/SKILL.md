# Tauri v2 Modular Architecture (React + Rust)

## Purpose

Enforce modular architecture in Tauri v2 desktop applications. Frontend: Vite + React + TanStack Router with file-based routing. Backend: Rust with layered architecture (commands → services → repositories → adapters). Both sides share the same modular discipline — replaceable modules, clear boundaries, no spaghetti.

---

# Core Principle

> A module is a self-contained business capability. On the frontend, it owns its Zod schemas, types, stores, hooks, UI, and DI wiring. On the Rust side, it owns its commands, services, repositories, and domain types. No other module may reach into its internals.

Every frontend module must be extractable into its own package **without rewriting business logic**. Every Rust module must be extractable into its own crate **without changing its trait contracts**. If you can't extract it cleanly, the boundary is broken.

# Architecture Overview

```
┌─────────────────────────────────────────────────────────┐
│  Tauri Desktop App                                      │
│                                                         │
│  ┌─────────────────────┐   ┌─────────────────────────┐  │
│  │  Frontend (Vite)     │   │  Backend (Rust)          │  │
│  │                      │   │                          │  │
│  │  React + TanStack    │   │  commands/               │  │
│  │  Router + shadcn     │   │    ↓ invoke()            │  │
│  │                      │   │  services/               │  │
│  │  modules/auth/        │   │    ↓ trait calls         │  │
│  │    ui/ login-form     │   │  repositories/           │  │
│  │    stores/ auth.store │   │    ↓ trait calls         │  │
│  │    application/       │   │  adapters/               │  │
│  │    repositories/ ─────┼───┤    → filesystem          │  │
│  │                      │   │    → external API         │  │
│  │                      │   │    → database             │  │
│  └─────────────────────┘   └─────────────────────────┘  │
│                                                         │
│  IPC: invoke() ── typed commands ── events/channels     │
└─────────────────────────────────────────────────────────┘
```

**Frontend calls Rust** via `@tauri-apps/api` `invoke()`. Frontend repositories wrap `invoke()` calls — not axios. External HTTP calls (to third-party APIs) go through the Rust backend for security, not from the browser.

**Rust never touches UI.** Commands return JSON. Frontend types match Rust response shapes.

## TanStack Router — File-Based Routes (Next.js-Style)

### Plugin Setup

```bash

```

```ts
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [
    TanStackRouterVite({
      routesDirectory: 'src/routes',
      generatedRouteTree: 'src/routeTree.gen.ts',

      quoteStyle: 'single',
      semicolons: true,

  resolve: {
    alias: { '@': path.resolve(__dirname, 'src') },
  },
```

### Router Creation

```tsx
// src/main.tsx
import { createRouter, RouterProvider } from '@tanstack/react-router';
import { routeTree } from './routeTree.gen';
import ReactDOM from 'react-dom/client';

const router = createRouter({ routeTree });

declare module '@tanstack/react-router' {
  interface Register {
    router: typeof router;
  }
}

const rootEl = document.getElementById('root');
if (rootEl) {
  const root = ReactDOM.createRoot(rootEl);
  root.render(<RouterProvider router={router} />);
}
```

### Route Rules

| Rule | Why |
|------|-----|
| `__root.tsx` is the only layout with providers | Single provider tree. No per-route provider nesting. |
| `_prefix` for pathless layout routes | `_authenticated/` groups routes without adding `/authenticated` to URL. |
| `$param` for dynamic segments | `$id.tsx` = `:id`. Type-safe — `Route.useParams()` returns typed `{ id: string }`. |
| `routeTree.gen.ts` is generated — never edit | Plugin reads `src/routes/` and generates the tree. Commit it. |
| Pages re-export module page components | Route file = thin shell: export route config + export default ModulePage. |
| Auth guards in `beforeLoad` | Route-level: `beforeLoad: ({ context }) => { if (!context.auth.user) throw redirect({ to: '/login' }) }` |

# Frontend — Module System

Same architecture as the Next.js skill. Replicated here for completeness.

## Dependency Direction

```
ui → stores/hooks → application → repositories

- schemas/  depends on nothing — pure Zod
- types/    depends on nothing — pure TypeScript interfaces
- stores/   depends on application/, types/ — factory receives services
- hooks/    depends on application/, types/ — wraps TanStack Query
- application/ depends on types/, schemas/, repositories/
- repositories/ depends on types/, shared (ILogger, Tauri IPC)
- ui/       depends on stores/, hooks/, schemas/
- di/       depends on everything — wires the module
```

## Module Internal Structure

```
src/modules/auth/
  schemas/          # Zod validation schemas — request shapes
  types/            # API response types, error types (plain interfaces)
  stores/           # Zustand stores — client state, factory receives services
  hooks/            # TanStack Query hooks — useQuery/useMutation wrappers
  application/      # Services — thin orchestration, calls repositories
  repositories/     # Data fetching — wraps invoke() to Rust backend
  ui/               # React components — pages, forms, widgets
  di/               # tsyringe child container + tokens
  index.ts          # Public API barrel
```

### `repositories/` — Tauri IPC Calls

In Tauri, repositories wrap `invoke()` instead of axios. Every Rust command becomes a typed async function.

```ts
// src/modules/auth/repositories/auth.repository.ts
import { inject, injectable } from 'tsyringe';
import { invoke } from '@tauri-apps/api/core';
import { SHARED_TOKENS } from '@/modules/shared';
import type { ILogger } from '@/modules/shared/types/logger';
import type { LoginResponse, UserDto } from '../types/responses';

@injectable()
export class AuthRepository {
  private readonly log: ILogger;

  constructor(@inject(SHARED_TOKENS.Logger) logger: ILogger) {
    this.log = logger.child({ module: 'auth', layer: 'repository' });
  }

  async login(email: string, password: string): Promise<LoginResponse> {
    this.log.debug({ command: 'auth_login' }, 'ipc call');
    try {
      return await invoke<LoginResponse>('auth_login', { email, password });
    } catch (err) {
      this.log.error({ command: 'auth_login', error: err }, 'ipc error');
      throw err;
    }
  }

  async getCurrentUser(): Promise<UserDto> {
    return invoke<UserDto>('get_current_user');
  }
}
```

### FormField & FieldError — Same Components

Identical to the Next.js skill. TanStack Form + Zod adapter. `FieldError` renders all errors. `FormField` wires TanStack Form's `field` object to shadcn-styled DOM.

```tsx
// src/modules/shared/ui/field-error.tsx
'use client'; // not needed in Vite, kept for consistency

import { cn } from '@/modules/shared/utils/cn';

interface FieldErrorProps {
  errors: unknown[] | undefined;
  className?: string;
}

export function FieldError({ errors, className }: FieldErrorProps) {
  if (!errors?.length) return null;

  return (
    <ul className={cn('text-xs text-destructive flex flex-col gap-0.5 mt-1', className)}>
      {errors.map((err, i) => {
        const message =
          typeof err === 'string'
            ? err
            : typeof err === 'object' && err !== null && 'message' in err
              ? (err as { message: string }).message
              : null;
        if (!message) return null;
        return (
          <li key={i} className="flex items-start gap-1">
            <span aria-hidden="true" className="select-none">&bull;</span>
            <span>{message}</span>
          </li>
        );
      })}
    </ul>
  );
}
```

```tsx
// src/modules/shared/ui/form-field.tsx
import type { ReactNode } from 'react';
import { cn } from '@/modules/shared/utils/cn';
import { FieldError } from './field-error';

  type = 'text',
  autoComplete,
  placeholder,
  disabled = false,
  rightLabel,
  rightElement,

  const hasErrors = field.state.meta.errors.length > 0;

  return (

          disabled={disabled}
          value={field.state.value}
          onChange={(e) => field.handleChange(e.target.value)}
          onBlur={field.handleBlur}

            'w-full h-9 rounded-md border bg-background px-3 py-1 text-sm text-foreground',

            rightElement && 'pr-10',
            className,
          )}
        />
        {rightElement}
      </div>

      <FieldError errors={field.state.meta.errors} />
    </div>
  );
}
```

### Logger — ILogger + Pino

Same ILogger interface. PinoLogger detects Tauri's webview runtime.

```ts
// src/modules/shared/infrastructure/pino-logger.ts
import pino, { type Logger as PinoInstance } from 'pino';
import type { ILogger } from '../types/logger';
import { env } from '@/env';

const isProd = env.NODE_ENV === 'production';

function createRootPino(): PinoInstance {
  return pino({
    level: isProd ? 'info' : 'debug',
    ...(isProd
      ? {}
      : {
          transport: {
            target: 'pino/browser',
            options: {
              transmit: {
                send: (_level: string, logEvent: { messages: unknown[] }) => {
                  const [obj, msg] = logEvent.messages;
                  const prefix = obj && typeof obj === 'object' && 'module' in obj
                    ? `[${(obj as Record<string, string>).module}]`
                    : '';
                  console.log(prefix, msg ?? obj);
                },
              },
            },
          },
        }),
    browser: { asObject: true },
  });
}

const rootPino = createRootPino();

export class PinoLogger implements ILogger {
  constructor(private readonly pino: PinoInstance) {}

  debug(obj: Record<string, unknown>, msg?: string): void { this.pino.debug(obj, msg); }
  info(obj: Record<string, unknown>, msg?: string): void { this.pino.info(obj, msg); }
  warn(obj: Record<string, unknown>, msg?: string): void { this.pino.warn(obj, msg); }
  error(obj: Record<string, unknown>, msg?: string): void { this.pino.error(obj, msg); }
  child(bindings: Record<string, unknown>): ILogger {
    return new PinoLogger(this.pino.child(bindings));
  }
}

export const rootLogger: ILogger = new PinoLogger(rootPino);
```

### Environment Variables — Vite + Rust (Split System)

Vite and Rust do **not** share a runtime. `.env` is a Vite-only convention. Rust reads env vars from the OS at process start. Two separate sources, one `.env.example` file for documentation.

#### `.env.example` — Single Template for Both

```bash
# .env.example — commit this. Copy to .env for dev.

# ── Vite Frontend (VITE_ prefix) ──────────────────
# Loaded by Vite at build/dev time. Bundled into the app.
VITE_APP_NAME=MyApp
VITE_LOG_LEVEL=debug

# ── Rust Backend (no prefix, runtime OS env) ──────
# Read by Rust at app launch via std::env::var(). Never bundled.
# Tauri injects these into the Rust process from the OS environment.
API_BASE_URL=http://localhost:8080/api/v1
DATABASE_URL=sqlite://app.db?mode=rwc
```

```ts
  isLoading: boolean;
  error: string | null;
  login: (email: string, password: string) => Promise<void>;
  logout: () => Promise<void>;
  setUser: (user: UserDto | null) => void;

    user: null,
    isLoading: false,
    error: null,

    login: async (email, password) => {
      set({ isLoading: true, error: null });
      try {
        const user = await authService.login({ email, password });
        set({ user, isLoading: false });
      } catch (err) {

      }
    },

    logout: async () => {
      await authService.logout();
      set({ user: null });
    },

    setUser: (user) => set({ user }),
  }));
}

export type AuthStore = ReturnType<typeof createAuthStore>;
```

# Single Instance — Prevent Multiple App Windows

By default, double-clicking the app icon launches a second process. Two windows, two state containers, shared data fighting over the same files — corruption risk.

`tauri-plugin-single-instance` enforces one app process. Second launch triggers a callback instead of spawning a new window. The callback can focus the existing window, restore it from tray, or process a deep-link argument.

### Cargo Dependency

```toml
[dependencies]
tauri-plugin-single-instance = "2"
```

### `lib.rs` — Setup

```rust
// src-tauri/src/lib.rs
#[cfg_attr(mobile, tauri::mobile_entry_point)]
pub fn run() {
    let config = AppConfig::from_env();

    tauri::Builder::default()
        .plugin(tauri_plugin_single_instance::init(|app, argv, _cwd| {
            // Second instance triggered — focus existing window instead
            if let Some(window) = app.get_webview_window("main") {
                let _ = window.show();
                let _ = window.set_focus();
            }
            // argv contains CLI args from the second launch attempt.
            // Use it for deep-linking: myapp://open/project/123 → argv[1]
        }))
        .setup(|app| {
            let services = di::build_services(&config, app.handle().clone());
            app.manage(AppState {
                auth_service: services.auth_service,
                dashboard_service: services.dashboard_service,
            });
            Ok(())
        })
        .invoke_handler(tauri::generate_handler![
            commands::auth_commands::auth_login,
            commands::auth_commands::get_current_user,
            commands::auth_commands::auth_logout,
        ])
        .run(tauri::generate_context!())
        .expect("error while running tauri application");
}
```

**How it works**: The plugin creates a mutex (OS-level named lock). First instance acquires it. Second instance detects it, fires the callback, then exits. The callback runs in the **first** instance's process — you call `window.show()` + `window.set_focus()` on the already-running window. ~10 lines of code total.

### Deep-Link Support (Optional)

Pass CLI args through to the frontend on second launch:

```rust
tauri_plugin_single_instance::init(|app, argv, _cwd| {
    if let Some(window) = app.get_webview_window("main") {
        let _ = window.show();
        let _ = window.set_focus();
    }
    // Forward deep-link to frontend as an event
    if argv.len() > 1 {
        let _ = app.emit("deep-link", argv[1].clone());
    }
})
```

Frontend listens: `listen<string>('deep-link', (event) => navigateTo(event.payload))`.

### Rules

| Rule | Why |
|------|-----|
| `window.show()` + `window.set_focus()` in callback | User double-clicks the icon — they expect the app to appear. Don't silently swallow the second click. |
| No second `tauri::Builder` or window creation in callback | Second instance exits. All logic runs in the first instance's callback. |
| Emit deep-link args as events | Frontend router handles navigation. Rust knows about URLs, not routes. |

# System Tray — Minimize to Tray Instead of Closing

Desktop apps should not die when the user clicks X. Close → hide to tray. App stays alive in the background. The tray gives users a way to quit for real.

Two Tauri plugins: `tauri-plugin-tray` (always needed) + `tauri-plugin-shell` / your own menu handler (for context menu actions).

### Cargo Dependencies

```toml
[dependencies]
tauri-plugin-tray = "2"
tauri-plugin-shell = "2"    # Already in deps — used for tray menu events
```

### Capability

```json
// src-tauri/capabilities/default.json
{
  "permissions": [
    "tray:default",
    "tray:allow-new",
    "tray:allow-set-icon",
    "tray:allow-set-tooltip",
    "tray:allow-set-show-menu-on-left-click"
  ]
}
```

### `lib.rs` — System Tray Setup

```rust
// src-tauri/src/lib.rs
use tauri::{
    tray::{MouseButton, MouseButtonState, TrayIconBuilder, TrayIconEvent},
    menu::{MenuBuilder, MenuItemBuilder},
    Manager,
};

pub fn run() {
    let config = AppConfig::from_env();

    tauri::Builder::default()
        .plugin(tauri_plugin_single_instance::init(|app, _, _| {
            if let Some(window) = app.get_webview_window("main") {
                let _ = window.show();
                let _ = window.set_focus();
            }
        }))
        .setup(|app| {
            let services = di::build_services(&config, app.handle().clone());
            app.manage(AppState { /* ... */ });

            // ── System Tray ──────────────────────────────────
            let show = MenuItemBuilder::with_id("show", "Show").build(app)?;
            let quit = MenuItemBuilder::with_id("quit", "Quit").build(app)?;
            let menu = MenuBuilder::new(app).items(&[&show, &quit]).build()?;

            let _tray = TrayIconBuilder::new()
                .icon(app.default_window_icon().unwrap().clone())
                .tooltip(&config.app_name)
                .menu(&menu)
                .show_menu_on_left_click(false) // left-click = show window, right-click = menu
                .on_menu_event(|app, event| match event.id().as_ref() {
                    "show" => {
                        if let Some(window) = app.get_webview_window("main") {
                            let _ = window.show();
                            let _ = window.set_focus();
                        }
                    }
                    "quit" => {
                        app.exit(0);
                    }
                    _ => {}
                })
                .on_tray_icon_event(|tray, event| {
                    // Left-click on tray icon → toggle window
                    if let TrayIconEvent::Click {
                        button: MouseButton::Left,
                        button_state: MouseButtonState::Up,
                        ..
                    } = event
                    {
                        let app = tray.app_handle();
                        if let Some(window) = app.get_webview_window("main") {
                            if window.is_visible().unwrap_or(false) {
                                let _ = window.hide();
                            } else {
                                let _ = window.show();
                                let _ = window.set_focus();
                            }
                        }
                    }
                })
                .build(app)?;

            Ok(())
        })
        // ── Close → Hide Instead of Quit ──────────────────
        .on_window_event(|window, event| {
            if let tauri::WindowEvent::CloseRequested { api, .. } = event {
                api.prevent_close();
                let _ = window.hide();
            }
        })
        .invoke_handler(tauri::generate_handler![
            // ...
        ])
        .run(tauri::generate_context!())
        .expect("error while running tauri application");
}
```

### Tray Icon

Put an `.ico` file in `src-tauri/icons/`. Tauri's `default_window_icon()` picks it up from `tauri.conf.json` → `app.icon`. For production, export at 16×16, 32×32, 64×64 — the OS scales per DPI.

### Rules

| Rule | Why |
|------|-----|
| `.on_window_event(CloseRequested)` with `api.prevent_close()` + `window.hide()` | X button hides, doesn't quit. User stays logged in, background tasks keep running. |
| Tray menu has "Quit" — call `app.exit(0)` | Only way to actually stop the app. No `Cmd+Q` / `Alt+F4` cheat. |
| Left-click tray icon toggles window (show/hide) | Feels native. Windows users expect this. macOS users expect right-click menu too. |
| `show_menu_on_left_click(false)` | Left-click = toggle window. Right-click = tray menu. Same behavior as Slack, Discord, Spotify. |
| `MenuItemBuilder` IDs: `"show"` + `"quit"` | Simple string matching in `on_menu_event`. No macro magic. |
| Tray icon from `default_window_icon()` | Uses `app.icon` from `tauri.conf.json`. One icon for window title bar AND tray. |

# WebView Runtime — Cross-Platform Compatibility

The "app opens blank/white screen on some machines" problem is always missing or incompatible WebView runtime. Tauri does **not** bundle a browser engine — it uses the OS's native one.

## Platform-by-Platform

| Platform | Engine | Minimum OS | Common Issue |
|----------|--------|-----------|-------------|
| Windows | WebView2 (Edge-based) | Windows 10 1809+ / Windows 11 | Not installed on LTSC, Enterprise, Server, stripped builds. **Windows 8.x and older unsupported.** |
| macOS | WKWebView | macOS 10.15+ | Built into every macOS 10.15+. No issues. |
| Linux | webkit2gtk | Any distro | Must be installed manually. Tauri v2 drops 32-bit Linux support. |

## Windows 8 / 8.1 — Not Supported

Tauri v2 requires WebView2. WebView2 requires Windows 10 1809 (October 2018 update) or newer. **Windows 8, 8.1, and Windows 7 cannot run Tauri v2 apps.** No workaround — WebView2's C++ runtime (`WebView2Loader.dll`) links against APIs that don't exist in those kernels. Tauri v1 had the same restriction for its final releases.

If you need Windows 8 support: use Electron (Chromium bundled, no OS WebView dependency) or a native UI framework (Qt, WinUI). Tauri trades OS compatibility for binary size.

## Windows 10 — Minimum Build Check

Windows 10 pre-1809 (LTSC 2016, LTSC 2019) also lack WebView2. The bundled fixed-version WebView2 fixes this IF the OS is 1809+. Below 1809, nothing works.

## Windows — WebView2 Missing

Windows 10/11 LTSC, Enterprise, Server, and debloated builds may not have the Evergreen WebView2 Runtime. App opens a blank white window.

**Fix**: Bundle a fixed-version WebView2 with the installer.

```json
// src-tauri/tauri.conf.json — tauri handles the rest
{
  "bundle": {
    "windows": {
      "webviewInstallMode": {
        "type": "fixedRuntime",
        "path": "./webview2"
      }
    }
  }
}
```

Download the fixed-version WebView2 runtime from Microsoft's [WebView2 downloads page](https://developer.microsoft.com/en-us/microsoft-edge/webview2/) (Fixed Version tab) into `src-tauri/webview2/`. Tauri embeds it in the MSI/NSIS installer — no internet, no preinstalled runtime, no admin rights needed. Adds ~150MB. One-time cost, zero user support tickets.

## Linux — webkit2gtk Missing

```bash
# Debian/Ubuntu
sudo apt install libwebkit2gtk-4.1-dev libgtk-3-dev libayatana-appindicator3-dev
# Fedora
sudo dnf install webkit2gtk4.1-devel gtk3-devel libappindicator-gtk3-devel
# Arch
sudo pacman -S webkit2gtk-4.1 libappindicator-gtk3
```

Document these as prerequisites in your app's README. AppImage/Flatpak can bundle `webkit2gtk`.

# Code Signing — Windows SmartScreen & macOS Gatekeeper

Unsigned desktop binaries trigger scary "this app might harm your device" warnings on every OS. Each platform has its own gatekeeper — none can be skipped. Users need a way to run the app without warnings.

## Windows — SmartScreen "Windows protected your PC"

Appears every time an unsigned `.exe` is downloaded from the internet. User must click "More info" → "Run anyway." Kills first-impression trust.

### Available Options

| Method | Cost | Removes Warning | Effort |
|--------|------|-----------------|--------|
| EV Code Signing Certificate | ~$300-400/year | Yes, immediately | Package cert with `tauri.conf.json` → `bundle.windows.signCommand` |
| OV Code Signing Certificate | ~$100-200/year | After reputation builds (weeks/months of downloads) | Same as EV but slower trust |
| Nothing (unsigned) | Free | Never | Users click "More info" → "Run anyway" forever |

### EV Signing with Tauri

```json
// src-tauri/tauri.conf.json
{
  "bundle": {
    "windows": {
      "signCommand": "signcode -t http://timestamp.digicert.com -a sha256 -spc %1.spc -v %1.pvk %1"
    }
  }
}
```

Tauri calls `signCommand` during `tauri build`. Replace with your CI's signing tool (Azure Key Vault, `signtool.exe`, etc.). EV certs use hardware tokens or HSM-backed cloud keys.

### The Reputation Trap

SmartScreen tracks download volume per certificate. New cert + low downloads = "uncommon" warning, even with valid signing. This takes ~3,000-10,000 downloads to clear. EV certs skip this check. OV certs don't.

### Recommendation

- Internal/corporate tools: skip signing. Users know they installed it.
- Public-facing app: EV cert from day one. The cost is the same across vendors (DigiCert, Sectigo, etc.).
- Open-source: file for Azure's free code signing via Trusted Signing or sign with GitHub Actions + `signtool`.

### Installer vs Portable

MSI/NSIS installers signed with an EV cert show the publisher name in the UAC dialog (blue shield). Unsigned installers show "Unknown publisher" in orange/red. This is separate from SmartScreen — happens before the app even launches. Same cert covers both.

## macOS — Gatekeeper "cannot be opened because it is from an unidentified developer"

Every unsigned `.app` triggers this. User must go to System Settings → Privacy & Security → "Open Anyway." Worse UX than Windows.

### Available Options

| Method | Cost | Removes Warning |
|--------|------|-----------------|
| Apple Developer Program + Notarization | $99/year | Yes |
| Ad-hoc signing | Free | No — still triggers Gatekeeper, `codesign` alone not enough |
| Nothing | Free | User must manually override every version |

### Notarization Flow

```
car tauri build (macOS)
  → codesign --sign "Developer ID Application: ..." MyApp.app
  → zip MyApp.app
  → xcrun notarytool submit MyApp.zip
  → xcrun stapler staple MyApp.app
  → ship .dmg
```

Tauri's `tauri.conf.json` supports `bundle.macOS.signingIdentity` for codesign. Notarization happens in CI (GitHub Actions, Xcode Cloud) after the `.dmg` is built. Notarization takes ~2-5 minutes per build.

### The DMG Gate

Even a notarized app, downloaded inside a `.dmg`, triggers a Gatekeeper dialog on first open saying "MyApp is an app downloaded from the Internet. Are you sure you want to open it?" This is unavoidable — macOS flags all files that land in Downloads with the `com.apple.quarantine` extended attribute. A `.pkg` installer avoids this.

## Recommendation

| Audience | Windows | macOS |
|----------|---------|-------|
| Internal tool | Skip | Skip |
| Public launch | EV cert | Apple Developer + Notarize |
| Open-source | GitHub Actions + Azure Trusted Signing | Ad-hoc + user manual override |

Start unsigned. Don't delay launch for code signing. Add signing once users exist — the warning doesn't prevent running the app, it just looks bad.

# Version Bumping — Where to Change

Three places. Everything else reads from these:

| File | Key | Scope |
|------|-----|-------|
| `src-tauri/tauri.conf.json` | `version` | Installer metadata, Windows file version, macOS bundle version |
| `src-tauri/Cargo.toml` | `package.version` | Rust crate version embedded in binary, `env!("CARGO_PKG_VERSION")` runtime value |
| `package.json` | `version` | Frontend version in devtools, About screen, `VITE_APP_VERSION` env var |

### Sync Rule

All three must match. If `tauri.conf.json` says `1.0.3` but `Cargo.toml` says `1.0.2`, the installer shows the wrong version. CI should assert this:

```json
// package.json — script
{
  "scripts": {
    "check-version": "bun run scripts/check-version.ts"
  }
}
```

Or simpler: only bump in CI. Your CI workflow reads one version tag, writes all three files, then builds. No human touches version numbers.

### Where Versions Are NOT Needed

- `tauri.conf.json` → `build.beforeBuildCommand` — never contains a version. Just `bun run build`.
- Frontend source code — never hardcode a version. Use `import.meta.env.VITE_APP_VERSION` (set by Vite's `define`) or read from `tauri.conf.json` at build time.
- `<title>` in `index.html` — static. Change only when the product name changes, not on every release.

## macOS

No action needed. WKWebView is part of the OS. Every Mac since 2014 ships it. Sandboxed App Store apps use it natively.

## Recommendation for Your Case

You're on Windows 11 Enterprise LTSC — the exact SKU where WebView2 is often absent. Use **Strategy 1 (fixedRuntime)**. One-time 150MB upfront cost, zero user support tickets about blank windows. Ship it in the installer via `tauri.conf.json`.

# App Name — Where to Change

One source of truth, consumed everywhere else:

| File | Key | Scope |
|------|-----|-------|
| `src-tauri/tauri.conf.json` | `productName` | Window title bar, OS taskbar, Start menu, About dialog. **The single source of truth.** |
| `src-tauri/tauri.conf.json` | `identifier` | Bundle ID: `com.mycompany.myapp`. Never changes after first release — identifier links to OS app data. |

### How Everything Else Reads It

**Rust side** — `tauri.conf.json` is baked into the binary at compile time. Tauri reads `productName` and `identifier` automatically for window titles, tray tooltips, and OS metadata. No manual wiring needed.

**Frontend side** — NOT from `tauri.conf.json`. Use T3 Env:

```bash
# .env
VITE_APP_NAME=MyApp
```

```ts
// src/modules/shared/ui/about-dialog.tsx
import { env } from '@/env';

export function AboutDialog() {
  return <span>{env.VITE_APP_NAME} v{import.meta.env.VITE_APP_VERSION}</span>;
}
```

### What Must Match

| Source | What |
|--------|------|
| `tauri.conf.json` → `productName` | Window title, installer name, OS metadata |
| `.env` → `VITE_APP_NAME` | Frontend `<title>`, About screen, login screen branding |
| `tauri.conf.json` → `identifier` | Reverse domain: `com.mycompany.myapp` |

`productName` in `tauri.conf.json` should equal `VITE_APP_NAME` in `.env`. Human-readable. Spaces allowed. Case matters. `identifier` is different — reverse domain, lowercase, no spaces, permanent.

Every Rust module follows four layers. Direction: **commands → services → repositories → adapters**. Each layer depends only on the layer below it and `domain/` types. Traits define contracts. Concrete implementations are wired in `di.rs`.

```
commands/      Thin IPC handlers — deserialize params, call service, serialize response
    ↓
services/      Business logic — trait-defined, orchestrates repositories
    ↓
repositories/  Data access — trait-defined, calls adapters
    ↓
adapters/      External I/O — filesystem, HTTP, database, system APIs
```

`domain/` has **zero dependencies** — pure structs, enums, error types. Everything else depends on it. It depends on nothing.

## `adapters/` — External I/O

Adapters wrap external dependencies. All implement traits defined in `repositories/` or are consumed directly by repository implementations.

### HTTP Client — reqwest + rustls (Critical for Cross-Platform TLS)

**The problem**: `reqwest::Client::new()` uses `native-tls` by default. On Windows, native-tls uses SChannel — it trusts only certificates in the Windows certificate store. Users with outdated root certs, enterprise MITM proxies, or stripped Windows installations get `certificate verify failed` / `SSL error:1416F086`. Some machines work, others don't — same binary, different OS trust stores.

**The fix**: Use `rustls-tls` with `webpki-roots`. This bundles Mozilla's root CA store inside the binary. Every machine trusts the same certificates — identical behavior on Windows, macOS, and Linux.

```toml
# Cargo.toml — override reqwest's default TLS
[dependencies]
reqwest = { version = "0.12", default-features = false, features = ["rustls-tls", "json"] }
```

```rust
// src-tauri/src/adapters/http_client.rs
use reqwest::Client;
use crate::domain::error::{AppError, AppResult};
use std::time::Duration;

pub struct HttpClient {
    client: Client,
    base_url: String,
}

impl HttpClient {
    pub fn new(base_url: String) -> Self {
        let client = Client::builder()
            .timeout(Duration::from_secs(30))
            .connect_timeout(Duration::from_secs(10))
            .user_agent(format!("{}/{}", env!("CARGO_PKG_NAME"), env!("CARGO_PKG_VERSION")))
            .build()
            .expect("Failed to build HTTP client");

        Self { client, base_url }
    }

    pub async fn get<T: serde::de::DeserializeOwned>(&self, path: &str) -> AppResult<T> {
        let url = format!("{}{}", self.base_url, path);
        let resp = self.client
            .get(&url)
            .send()
            .await
            .map_err(|e| AppError::ExternalApi(e.to_string()))?;

        if !resp.status().is_success() {
            return Err(AppError::ExternalApi(format!("HTTP {}", resp.status())));
        }

        resp.json().await.map_err(|e| AppError::ExternalApi(e.to_string()))
    }

    pub async fn post<T: serde::de::DeserializeOwned>(
        &self,
        path: &str,
        body: &impl serde::Serialize,
    ) -> AppResult<T> {
        let url = format!("{}{}", self.base_url, path);
        let resp = self.client
            .post(&url)
            .json(body)
            .send()
            .await
            .map_err(|e| AppError::ExternalApi(e.to_string()))?;
        resp.json().await.map_err(|e| AppError::ExternalApi(e.to_string()))
    }
}
```

**Why each setting**:

| Choice | Why |
|--------|-----|
| `default-features = false, features = ["rustls-tls"]` | Bundles Mozilla root CAs. No dependency on OS cert store. Identical TLS on every machine. |
| `timeout(30s)` | No request hangs forever. `reqwest` has no default timeout — without this, a dead server blocks the thread. |
| `connect_timeout(10s)` | Separate timeout for TCP/TLS handshake. Faster fail on unreachable hosts. |
| `user_agent(...)` | Many APIs (GitHub, Cloudflare) reject requests without a user-agent. Identify your app. |
| `Client::builder()` over `Client::new()` | Explicit config. `Client::new()` hides all defaults — you don't know what you're shipping. |

**No custom `rustls::ClientConfig` needed** — reqwest's `rustls-tls` feature depends on `webpki-roots` automatically, which compiles Mozilla's root CA store into the binary at build time. No `.pem` files, no environment variables, no manual `rustls::ClientConfig`. This handles GitHub, Stripe, and any public HTTPS endpoint. Only build a custom config if you need mutual TLS (client certificates) or internal CAs.

## `repositories/` — Trait-Defined Data Access

```rust
// src-tauri/src/repositories/mod.rs
pub mod auth_repository;

// src-tauri/src/repositories/auth_repository.rs
use async_trait::async_trait;
use crate::domain::auth::{User, LoginRequest, LoginResponse};
use crate::domain::error::AppResult;

/// Repository trait — what, not how.
/// Implementations live in the same module or adapters/.
#[async_trait]
pub trait AuthRepository: Send + Sync {
    async fn login(&self, req: &LoginRequest) -> AppResult<LoginResponse>;
    async fn get_user_by_id(&self, id: &str) -> AppResult<User>;
    async fn save_tokens(&self, access: &str, refresh: &str) -> AppResult<()>;
    async fn clear_tokens(&self) -> AppResult<()>;
}

/// Concrete implementation using HTTP adapter
pub struct AuthRepositoryImpl {
    http: crate::adapters::http_client::HttpClient,
    storage: crate::adapters::file_storage::FileStorage,
}

impl AuthRepositoryImpl {
    pub fn new(
        http: crate::adapters::http_client::HttpClient,
        storage: crate::adapters::file_storage::FileStorage,
    ) -> Self {
        Self { http, storage }
    }
}

#[async_trait]
impl AuthRepository for AuthRepositoryImpl {
    async fn login(&self, req: &LoginRequest) -> AppResult<LoginResponse> {
        self.http.post("/auth/login", req).await
    }

    async fn get_user_by_id(&self, id: &str) -> AppResult<User> {
        self.http.get(&format!("/users/{}", id)).await
    }

    async fn save_tokens(&self, access: &str, refresh: &str) -> AppResult<()> {
        self.storage.write_json("tokens.json", &serde_json::json!({
            "access_token": access,
            "refresh_token": refresh,
        }))
    }

    async fn clear_tokens(&self) -> AppResult<()> {
        self.storage.delete("tokens.json")
    }
}
```

### Why `async_trait`

Native async traits in Rust are stable (1.75+), but `Send + Sync` bounds for Tauri's multi-threaded executor still benefit from `async_trait`'s explicit `Send + Sync` declaration. Prefer `async_trait` for repository and service traits — safer threading guarantees.

### SOLID in Services

| Principle | Rust Translation |
|-----------|-----------------|
| **S**ingle Responsibility | One service trait per domain concept. `AuthService` doesn't know about `OrderService`. |

### Command Rules

| Rule | Why |
|------|-----|
| Commands are thin — deserialize, delegate, return | Business logic in services, not commands. |
| Return `AppResult<T>` | Frontend expects `{ data?: T, error?: { kind, message } }`. |
| All params are owned types (`String`, not `&str`) | Async commands can't borrow from the IPC payload. |
| One command file per domain | `auth_commands.rs`, `order_commands.rs` — not one giant file. |
| Commands are `pub fn` in a module | Registered via `generate_handler![commands::auth_commands::auth_login, ...]`. |
| No `State<AppState>` access to internals directly | Commands only call service traits — never reach into repositories or adapters. |

# SOLID in Rust — Quick Reference

| Principle | Rust Implementation |
|-----------|-------------------|
| **S** — Single Responsibility | One trait per domain concept. One struct impl per trait. |

# Error Handling — `thiserror` + `AppResult<T>`

```rust
// Always use AppResult<T> as return type for fallible operations.
// Never return Result<T, Box<dyn Error>> — caller can't match on it.
// Never return Result<T, String> — no discriminant, frontend can't handle.

pub type AppResult<T> = Result<T, AppError>;

#[derive(Error, Debug, Serialize)]
#[serde(tag = "kind", content = "message")]
pub enum AppError {
    #[error("Not found: {0}")]
    NotFound(String),
    #[error("Validation error: {0}")]
    Validation(String),
    #[error("Unauthorized: {0}")]
    Unauthorized(String),
    #[error("Internal: {0}")]
    Internal(String),
}
```

**Why `#[serde(tag = "kind")]`**: Serializes to `{"kind": "NotFound", "message": "User 123 not found"}`. Frontend can switch on `kind` for typed error handling — no regex-parsing error strings.

# System Notifications — tauri-plugin-notification

Tauri v2 supports OS-native desktop notifications via `tauri-plugin-notification`. Two layers: Rust service (sends OS toast) + frontend store (app-internal flash messages). They serve different purposes.

## Frontend Store — Toast/Banner Notifications (Zustand)

App-internal UI feedback. No OS permission needed. Stays inside the app window.

```ts
// src/modules/shared/stores/notification.store.ts
import { create } from 'zustand';

export type NotificationType = 'success' | 'error' | 'warning' | 'info';

export interface Notification {
  id: string;
  type: NotificationType;
  title: string;
  message?: string;
  duration?: number; // ms — auto-dismiss. undefined = manual dismiss
}

interface NotificationState {
  notifications: Notification[];
  push: (n: Omit<Notification, 'id'>) => void;
  dismiss: (id: string) => void;
  clearAll: () => void;
}

let nextId = 0;

export const useNotificationStore = create<NotificationState>((set) => ({
  notifications: [],

  push: (n) => {
    const id = `notif-${++nextId}`;
    set((s) => ({ notifications: [...s.notifications, { ...n, id }] }));

    if (n.duration) {
      setTimeout(() => {
        set((s) => ({ notifications: s.notifications.filter((x) => x.id !== id) }));
      }, n.duration);
    }
  },

  dismiss: (id) => set((s) => ({ notifications: s.notifications.filter((n) => n.id !== id) })),
  clearAll: () => set({ notifications: [] }),
}));
```

**Frontend usage**: Any component calls `useNotificationStore.getState().push({ type: 'error', title: 'Login failed', message: '...', duration: 5000 })`. A shared `ToastContainer` component in `__root.tsx` renders the stack.

### Permission Request (Required on macOS/Linux)

macOS requires user opt-in before showing notifications. Windows doesn't. Always check first.

### Notification Adapter — Wraps the Plugin

```rust
// src-tauri/src/adapters/notification.rs
use tauri::AppHandle;
use tauri_plugin_notification::{NotificationExt, PermissionState};

pub struct NotificationAdapter {
    app: AppHandle,
}

impl NotificationAdapter {
    pub fn new(app: AppHandle) -> Self {
        Self { app }
    }

    /// Request permission if not already granted. Windows always returns Granted.
    pub fn ensure_permission(&self) -> bool {
        match self.app.notification().permission_state() {
            PermissionState::Granted => true,
            PermissionState::Denied => false,
            PermissionState::NotDetermined => {
                // PermissionState::NotDetermined — request
                // Note: tauri-plugin-notification shows the OS dialog on first notify() call.
                // No explicit request_permission() API in v2. Call notify() — it triggers the prompt.
                true
            }
        }
    }

    pub fn send(&self, title: &str, body: Option<&str>) {
        if !self.ensure_permission() {
            return;
        }
        self.app
            .notification()
            .builder()
            .title(title)
            .body(body.unwrap_or(""))
            .show()
            .ok();
    }
}
```

### Wire into Services (Not Commands)

Services send notifications as a side effect. Never expose raw `NotificationAdapter` to the frontend.

```rust
// src-tauri/src/services/order_service.rs (excerpt)
pub struct OrderServiceImpl<R: OrderRepository> {
    repo: R,
    notifier: NotificationAdapter,
}

#[async_trait]
impl<R: OrderRepository + Send + Sync> OrderService for OrderServiceImpl<R> {
    async fn create_order(&self, req: CreateOrderRequest) -> AppResult<Order> {
        let order = self.repo.create(&req).await?;
        self.notifier.send(
            "Order Created",
            Some(&format!("Order #{} has been placed", order.id)),
        );
        Ok(order)
    }
}
```

**Why notification adapter, not direct plugin call in service**: Testable — mock the adapter in unit tests without spawning a real notification. Swappable — swap OS notifications for a no-op adapter in CI or headless environments.

```rust
// Pass AppHandle to build_services so notifications can access the app
```

## Notification Rules

| Rule | Why |
|------|-----|
| `notification.store.ts` for in-app toasts | Zustand. No permission. Instant. Rendered as UI in the window. |
| `NotificationAdapter` for OS-native | Wraps tauri-plugin-notification. Service sends, user sees in OS tray. |
| Always check `permission_state()` before `show()` | macOS blocks if denied. Silent no-op better than panic. |
| Notification adapter is a service dependency | Injected via `di.rs`. Never called from commands directly. |
| Never expose `notification::show()` as a Tauri command | Frontend doesn't control OS notifications. Backend decides when to notify based on business events. |
| `AppHandle` passed via `setup()` not `manage()` | Notification plugin needs the app handle at construction time. `setup()` fires once after plugins init.

# Quick Rules Reference

| Rule | Why |
|------|-----|
| `src/routes/` is a thin routing shell | No business logic. Compose from modules. |
| `src-tauri/src/commands/` are thin IPC handlers | Delegate to services. No business logic. |
| `domain/` depends on nothing | Pure structs + errors. serde derives only. |
| `services/` depend on `domain/` + repository traits | Business logic only. No I/O. |
| `repositories/` depend on `domain/` + adapters | Data access contracts. Trait-defined. |
| `adapters/` depend on external crates | HTTP, filesystem, database. Swappable. |
| `di.rs` wires concretions | Only place that knows concrete types. |
| All services are `Arc<dyn Trait>` in `AppState` | Dependency inversion. Commands see traits, not structs. |
| Return `AppResult<T>` from every fallible operation | Frontend gets typed `{ data, error }`. |
| `#[serde(tag = "kind")]` on `AppError` | Discriminated union. Switch on `kind` in frontend. |
| `generate_handler![]` lists every command | Tauri requirement. One registration point. |
| Capabilities: least privilege | Only enable plugins the app actually uses. |
| Frontend modules mirror Next.js skill patterns | DI, Zod, Zustand, TanStack Query, FormField, Logger. |
| Zod validates ALL rules at once | No `.abortEarly`. User sees every error on first submit. |
| FormField shows ALL errors per field | Bullet list. Destructive border on input. |
| `ILogger` interface, PinoLogger implementation | Modules depend on trait. Structured logging always. |
| `useResolve` wraps `container.resolve` | Stable across renders via `useMemo`. |
| Child DI containers per module | Isolate registrations. Enforce boundaries. |
| Auth guards in `beforeLoad` | Route-level, not component-level. Redirect, don't render. |

# Anti-Patterns

| Anti-pattern | Correct |
|-------------|---------|
| Business logic in `src/routes/page.tsx` | Page re-exports module's page component |
| Business logic in `#[tauri::command]` | Command delegates to service method |
| `invoke()` directly in a component | Repository wraps invoke. Component calls store/hook. |
| Service calling `State<AppState>` directly | Service receives deps via constructor. `AppState` lives in `lib.rs`. |
| `Result<T, String>` in Rust | `AppResult<T>` with typed `AppError` |
| `Box<dyn Error>` as return type | `AppResult<T>` — caller needs to match on error kind |
| `Arc<dyn Trait>` for 2-method traits with trivial impls | Use concrete types when test substitution isn't needed |
| Monolithic `services.rs` with all business logic | One service file per domain |
| Raw `invoke()` in store/service | Repository wraps IPC — typed, logged, error-normalized |
| `console.log` in services/repos | Inject `ILogger`, call `this.log.info({...}, msg)` |
| Stores in `ui/stores/` | `stores/` at module root — own layer |
| Hooks in `ui/hooks/` | `hooks/` at module root — own adapter layer |
| Importing another module's `repositories/` | Use barrel `index.ts` + DI token |
| `csp: null` in tauri.conf.json | Always set CSP explicitly |
| All plugins in capabilities "just in case" | Audit — only enabled plugins |

# Enforcement

When building Tauri v2 apps with modular architecture:

1. Every frontend feature starts as a module — `src/modules/<name>/`.
2. Every frontend module has `schemas/`, `types/`, `stores/`, `application/`, `repositories/`, `ui/`, `di/` — skip only if genuinely empty.
3. Every component is `'use client'`. No server rendering in Tauri.
4. Every route in `src/routes/` is a thin shell: route config + `export default ModulePage`.
5. Every IPC call goes through a repository — never raw `invoke()` in components/stores.
6. Every Rust command delegates to a service — never business logic in `#[tauri::command]`.
7. Every Rust service is a trait + implementation — dependency inversion via `Arc<dyn Trait>` in `AppState`.
8. Every Rust repository is a trait + implementation — swappable data access.
9. Every Rust `Result` uses `AppResult<T>` — typed error discriminant for frontend.
10. Every Rust error uses `thiserror` derive + `#[serde(tag = "kind")]` — structured, not stringified.
11. Every concrete dependency is wired in `di.rs` — nowhere else.
12. CSP is set in `tauri.conf.json` — `default-src 'self'` minimum.
13. Capabilities use least privilege — only plugins the app needs.
14. Every form validates with Zod before calling the service — TanStack Form `validators.onSubmit`.
15. Every form field uses `FormField` component — renders ALL errors, not just first.
16. Every log call passes structured object: `log.info({ userId }, 'msg')`.
17. When in doubt, apply the extraction test: "Can I move this module to a separate crate/package without rewriting its business logic?"
